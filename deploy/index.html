Production

job-platform-web

Static SPA built with Vite. The API URL is baked in at build time via environment variables.

Build

Set the public API base URL (no trailing slash), then build:

# .env.production or CI env
export VITE_API_BASE_URL=https://api.yourdomain.com

npm ci
npm run build

Output lives in dist/. Deploy those files to any static host or object storage + CDN.

VITE_* variables are inlined when you run vite build. Changing the API URL requires a new build, not just a host restart.

Docker (nginx + static dist)

The repo includes a multi-stage Dockerfile: Node builds the app; nginx serves /usr/share/nginx/html with SPA fallback (docker/nginx.conf). The container listens on port 3008. 

docker build \
  --build-arg VITE_API_BASE_URL=https://api.yourdomain.com \
  -t job-platform-web .

docker run --rm -p 8080:3008 job-platform-web

Open http://localhost:8080. Production builds must pass a real API URL as VITE_API_BASE_URL.

Dokploy — Dockerfile application

Use a Dockerfile deployment for this repo (not Docker Compose). Follow these steps in order:

  1. In Dokploy, create a new application and choose deployment type Dockerfile (build from Git).
  2. Point the application at this repository (job-platform-web). Dockerfile path: default Dockerfile at repo root unless your layout differs.
  3. Under build settings, add a build argument: VITE_API_BASE_URL = your public API base URL — no trailing slash. If visitors load this UI over HTTPS, this value must start with https:// (e.g. https://api.yourdomain.com). HTTP API URLs are blocked by the browser (mixed content). Change it → rebuild/redeploy the web image.
  4. Configure routing for your UI domain: expose / forward to container port 3008 (nginx listens on 3008 inside the image).
  5. On the API deployment (job-platform-api), set environment variable APP_BASE_URL to your exact frontend origin — the same scheme, host, and port users see in the browser for this UI (e.g. https://app.yourdomain.com). That keeps CORS and session cookies aligned with the SPA.

Redeploy the web image whenever the API URL changes — VITE_* is fixed at image build time.

HTTPS checklist

  1. Put TLS on your API host in Dokploy (domain + Let’s Encrypt on the api service, port 4000).
  2. Set API env API_PUBLIC_URL to that same public HTTPS URL (no trailing slash). Set APP_BASE_URL to your SPA’s HTTPS origin (include :3008 if it appears in the browser bar).
  3. Rebuild the SPA with build arg VITE_API_BASE_URL=https://… matching that API URL — then redeploy only after the API responds over HTTPS (try curl -I https://api…).

502 Bad Gateway

  • Wrong port: In Dokploy’s domain settings, the container port must be 3008 (where nginx listens). Not 5173, 80, or a random mapped port unless that map targets 3008.
  • Traefik network: If the app container is not reachable from Traefik (same symptom as the API Compose stack), ensure it is attached to dokploy-network or add the label traefik.docker.network=dokploy-network in Dokploy’s labels UI if your version supports it — see API deploy/index.html for context.
  • Build failed: Open deploy logs; if the Vite build failed, the image may be invalid or nginx has nothing to serve.

Align with the API

  • On the API, set APP_BASE_URL to this site’s exact origin (e.g. https://app.yourdomain.com). Cookies and CORS depend on it.
  • Use HTTPS in production for both SPA and API when handling sessions.
  • If you use preview URLs (e.g. Vercel previews), add them to the API’s CORS_ORIGINS or temporarily widen policy — see API README.

Hosting options

  • Dokploy: see Dokploy — Dockerfile application above.
  • Docker elsewhere: same image; pass VITE_API_BASE_URL at build time.
  • Nginx / Caddy: serve dist with SPA fallback to index.html for client-side routes.
  • Same origin: you can put the API behind a reverse proxy at /api — then you would point the SPA at that origin; today the client uses absolute VITE_API_BASE_URL (configure accordingly).
